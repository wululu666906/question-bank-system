# 智能题库自动生成系统 —— 项目需求文档 (PRD)

## 📌 1. 项目概述
### 1.1 产品愿景
打造一款面向教育培训、职业考试及企业内训的高质量“**流式智能题库生成器**”。通过前沿的 AI 工作流编排引擎（Dify）结合业界强大的防爬虫互联网内容检索能力，实现“给出一个职业与考点，就能在秒级延迟内‘活写’出一套拥有权威行业背景和深度解析的实操试题”。

### 1.2 核心痛点与解决方案
- **痛点一：传统大模型满嘴跑火车（幻觉）**。要选择题给判断题。
  - **解决**：在发送给大模型前植入 0 延迟的代码级格式枷锁（“要是敢乱给题型后果自负”），在源头消除格式幻觉。
- **痛点二：大模型的知识库滞后、干瘪**。出题缺乏现代职场的实战案例。
  - **解决**：外挂 Google 搜索 + Firecrawl 穿透爬虫，无视网站 5 秒盾强行拉取当季最新的全网白皮书作为出题的原材料。
- **痛点三：用户等待 AI 响应时间过长**，容易导致流失。
  - **解决**：前端与工作流引擎建立 Server-Sent Events (SSE) 单向长连接，大模型想到第一个字就推向前端页面，实现毫无顿挫的“打字机”体验。

## 💻 2. 前端需求 (Frontend)
### 2.1 页面交互规划
前端页面仅需设计为一个极其清爽的**“操作台 + 结果展板”**。

#### A. 试卷操作台 (配置区)
- **职业输入 (profession)**：带联想建议的文本框（例如：建筑注册安全工程师 / 临床医学主任医师）。
- **知识点输入 (topic)**：支持长文本输入的 Textarea，越具体越好。
- **难度选择 (difficulty)**：Dropdown（简单 / 中等 / 困难 / 专家地狱级）。
- **题型选择 (question_type)**：Checkbox 或 Select（单选 / 多选 / 判断 / 简答）。
- **生成数量 (question_count)**：数字输入框 (建议 1 ~ 10)。
- **一键生成按钮 (Generate)**：点击后需带有动效，触发“准备中...”状态。

#### B. 成绩展板 (呈现区)
- **骨架屏过渡**：点击生成后，在网络与爬虫运作的最初 2-3 秒，展示脉冲式的骨架屏 (Skeleton)，缓解等待焦虑。
- **流式打字机效果**：接收到 SSE 首个报文后，骨架屏消失，渲染引擎（如 `react-markdown` 或 `marked.js`）开始逐字、逐段地实时渲染传回的题目。
- **工具栏**：在推流结束后，展板右上角亮起【复制全文】、【一键导出为 Word】、【重新生成】等操作按钮。

### 2.2 前端接口与对接逻辑
由于后端由 Dify 托管，前端仅需接入 Dify OpenAPI：
1. **Endpoint**: `POST https://api.dify.ai/v1/workflows/run` （或者独立搭建的后端中转分发路由）。
2. **Payload 结构**:
   ```json
   {
       "inputs": {
           "profession": "法务律师",
           "topic": "劳动合同法辞退补偿计算",
           "difficulty": "专家地狱级",
           "question_type": "单选题",
           "question_count": "5"
       },
       "response_mode": "streaming",
       "user": "uid_8888"
   }
   ```
3. **数据解析**：拦截 HTTP 流式拆包（如使用原生 `fetch` 结合 `ReadableStream`），提取 `data: {"event": "text_chunk", "data": {"text": "【"}}` 中的 `text` 字段，追加到页面的状态里。

## ⚙️ 3. 智能体工作流需求 (Dify Backend Workflow)
后端逻辑承载在预编排好的 DAG 工作流中。

### 3.1 核心管线拓扑
采取“分叉并发”和“强制截流”的策略：

1. **[Start: 统一入口]**：接管前端打过来的 5 个 `inputs` 参数。
2. **[并发一：左路纪律委]**：一个极其轻量的 Python 代码节点。它会根据 `question_type`，瞬间输出一段带红线警告的系统 Prompt（如：“此任务必须且只能写选择题”）。
3. **[并发二：右路采编队]**：
   - 首先利用 `Google Search` 输入“职业+知识点”，在全球索引中搜寻出排名并关联度最高的页面。
   - 紧接着过一道正则提取代码节点，剥离出那排在第一的、纯粹无污染的目标链接 (`target_url`)。
   - 最后交由 `Firecrawl Scrape`，作为当前市面上最强大的反反爬虫插件，无视任何 Cloudflare 遮蔽，直接将整个目标网页“生吞”，返回提取好的干净长 Markdown 原文。
4. **[收水总汇：大模型首字直飞]**：
   将左路的“极其刻薄的纪律规则”与右路的“原汁原味的长文干货资料”共同交给 `DeepSeek-V3` 推理大模型。
   大模型吸收后开始出卷。工作流不设置任何后处理（如 JSON 格式化工具等，因为后处理会导致流式断裂），大模型想到什么字，Dify 的 `End` 节点就直接将这个字射出，推到前端。

## 🚀 4. 非功能性需求与性能指标
- **首字响应延迟 (TTFB)**：由于最耗时的 Google 搜索和 爬虫 需要一定时间，设定首个字符到达前端屏幕的容忍时间必须控制在 **<= 5 秒**，以保证不打断用户心流。
- **流式丝滑度**：生成过程不得出现段落式的“突然蹦出一大块”，需要维持打字机的匀速感。
- **无数据兜底 (Fallback)**：如果由于极端情况右路爬虫失败（如对方网站彻底崩塌），工作流设计需在提取 URL 时设有预设兜底机制（如默认定向至专业维基文献），确保整体出卷业务永不中断。
- **Token 消耗预期**：通过让工作流前端的提取脚本和爬虫完成“脏活累活”，核心 72B 大模型处理的是高度清洗的纯享版上下文，极大程度压减无用 Token 的消耗。
